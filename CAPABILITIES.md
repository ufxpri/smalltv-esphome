# SmallTV-Ultra (ESP8266) — 능력 범위와 한계

핵심: **ESP8266은 코어 1개.** 화면/무거운 코드가 CPU를 오래 잡으면 WiFi가 굶어 먹통이 된다.
그래서 이 기기는 **"천천히 갱신되는 예쁜 정보 화면"에 최적, "부드러운 실시간 그래픽"엔 부적합**이다.

## 하드웨어 봉투 (넘을 수 없는 벽)

| 자원 | 실제 값 | 의미 |
|---|---|---|
| CPU | ESP8266 단일코어 80/160MHz, **FPU 없음** (소수점 SW 연산·느림) | 무거운 연산·물리·3D 불가 |
| 여유 RAM | 전체 80KB 중 **~40KB 가용** (WiFi+웹+API가 나머지 사용) | 가장 빡빡한 벽. 풀프레임 버퍼·큰 데이터 불가 |
| 화면 대역폭 | SPI 40MHz(5MB/s). 풀프레임 115KB 전송만 ~23ms + 그리기 CPU | 풀스크린 갱신 **실질 1~5 fps** |
| WiFi 세금 | 매 순간 CPU 양보 필요 | 오래 잡는 연산 금지 (→ `wifi.connected` 게이팅) |
| 플래시 | 4MB (앱은 1MB 레이아웃 사용, LittleFS로 ~3MB 에셋 가능) | 이미지/GIF 저장 여지 있음 |
| 오디오 | 스피커/DAC 없음 | 소리 불가 |

## 🟢 잘 되는 것 (스위트스팟)
- **정보 대시보드**: 시계·날짜·날씨·달력·주식/코인 티커·교통 도착정보 (API JSON → 렌더)
- **Home Assistant 완전 연동**: 센서값 표시, 알림 표시기, HA에서 화면·백라이트·버튼 제어, 자동화 트리거
- **가벼운 애니메이션**: 파티클(비/눈), 시계바늘, 게이지, 스크롤 텍스트, 날씨별 배경 (~5fps, 작은 영역은 더 빠름)
- **이미지/슬라이드쇼**: 플래시(LittleFS)나 HTTP로 PNG/GIF 받아 저fps 표시 (순정이 하던 것)
- **알림/상태판**: REST·HA로 메시지 푸시

## 🟡 되긴 하나 조심 (경계선)
- 240×240 이미지 디코드: RAM·CPU 부담 → 한 번에 하나, 느림
- 아주 단순한 게임(스네이크·퐁 수준 저fps·저모션): 가능하나 끊김
- 다중 API 동시 폴링: 짧게 나눠서, 동기 대기 길면 WiFi 위협

## 🔴 안 되는 것 (한계선)
- 부드러운 30~60fps 풀스크린 애니메이션·동영상 ❌
- 제대로 된 실시간 게임(플랫포머·액션) ❌
- 3D·복잡한 물리·무거운 수학 ❌ (FPU 없음)
- 큰 데이터·다중 대형 이미지 동시 처리 ❌ (40KB 벽)
- 오디오 ❌

## 더 원하면: 하드웨어 지렛대 = ESP32
"리치/살아있는/화려한/게임" 티어는 ESP8266이 못 낸다. 그 이상은 **ESP32**로:
- 듀얼코어 240MHz + 520KB RAM(+PSRAM) + 하드웨어 FPU
- → LVGL 부드러운 UI, 풀프레임 더블버퍼, GIF/이미지, MJPEG 스트리밍, 가벼운 게임까지
- SmallTV **PRO** 등 ESP32 변종, 또는 이 케이스에 ESP32 보드 이식 / 별도 ESP32+디스플레이 경로

**요약**: 이 보드로는 "세련되게 움직이는 정보 화면"이 천장. 게임/화려함이 목표면 ESP32로 갈 것.
